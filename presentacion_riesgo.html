Arquitectura

Separación de Responsabilidades

El proyecto adopta un modelo cliente-servidor con lógica desacoplada:

  • Backend (FastAPI): cálculos científicos y validación de datos.
  • Frontend (Streamlit): interfaz de usuario e interactividad.
  • Models: Pydantic actúa como "aduana" de datos.
  • Services: lógica matemática encapsulada en clases especializadas.

El frontend jamás instancia clases de negocio — todo fluye por HTTP/JSON validado.

Decoradores

Decoradores FastAPI: Registradores de Rutas

Los decoradores @router.post, @router.get, etc., transforman funciones en endpoints HTTP.

  • @router.get(path): obtiene recursos (idempotente).
  • @router.post(path): crea/procesa recursos (no idempotente).
  • response_model: schema de respuesta (validación + OpenAPI).
  • async def: el decorador preserve el event loop — no bloques.

FastAPI auto-genera documentación en /docs (Swagger) y /redoc a partir de los decoradores.

Pydantic

Aduana de Datos: Validación Antes del Cálculo

Los modelos Pydantic en models.py actúan como filtro transaccional: invalidan payloads inconsistentes antes de tocar la lógica matemática.

@field_validator

Valida campos individuales. Se ejecuta antes que la validación de modelo completo.

Casos de uso:

  • Normalizar tickers a mayúsculas
  • Verificar que pesos sean ≥ 0
  • Trim de espacios en strings

@model_validator(mode='after')

Valida la consistencia entre campos, después de que todos fueron procesados.

Casos de uso:

  • len(tickers) == len(pesos)
  • ∑ pesos = 1.0 ± 0.01
  • Lógica transversal
models.py - Validadores de VaRRequest 
@field_validator("tickers")
@classmethod
def tickers_en_mayusculas(cls, v: list[str]) -> list[str]:
    return [t.upper().strip() for t in v]  # Normaliza

@field_validator("pesos")
@classmethod
def pesos_positivos(cls, v: list[float]) -> list[float]:
    if any(p < 0 for p in v):
        raise ValueError("Todos los pesos ≥ 0")
    return v

@model_validator(mode="after")
def validar_consistencia_portafolio(self) -> "VaRRequest":
    if len(self.tickers) != len(self.pesos):
        raise ValueError("len(tickers) != len(pesos)")
    suma = sum(self.pesos)
    if abs(suma - 1.0) > 0.01:
        raise ValueError(f"∑w={suma:.4f}, debe ser 1.0")
    return self
Async/Await

Programación Asíncrona en FastAPI

Las funciones endpoint son async def, lo que permite no bloquear el servidor mientras esperamos I/O (yfinance, FRED).

  • async def: cede el CPU a otras tareas mientras espera I/O.
  • await: punto donde se suspende y cede.
  • Beneficio: múltiples clientes simultáneos sin threads.

En nuestro caso: mientras el hilo A descarga datos de yfinance, el hilo B sigue procesando cálculos de CAPM para otro usuario.

Nota: En FastAPI, el async de yfinance y pandas detecta automáticamente si el código es CPU-bound vs I/O-bound.

DataService

Extracción e Higiene de Datos

DataService en services.py encapsula todas las llamadas a yfinance y aplica limpieza automática.

  • yfinance: descarga OHLCV desde Yahoo Finance.
  • auto_adjust=True: precios ajustados por splits y dividendos.
  • pandas: manejo de DataFrames, ffill() para gaps.
  • numpy: cálculos vectorizados de rendimientos log.

Rendimiento log: $r_t = \ln(P_t / P_{t-1})$ en lugar de simple. Propiedades mejor: aditividad temporal, estacionaridad.

services.py - DataService.get_rendimientos() 
def get_rendimientos(
    self, ticker: str, fecha_inicio=None, fecha_fin=None
) -> pd.DataFrame:
    """Calcula rendimientos log a partir de precios."""
    df_precios = self.get_precios(ticker, fecha_inicio, fecha_fin)
    cierre = df_precios["cierre"]
    
    # Rendimientos simples: R_t = (P_t - P_{t-1}) / P_{t-1}
    rend_simple = cierre.pct_change()
    
    # Rendimientos logarítmicos: r_t = ln(P_t / P_{t-1})
    rend_log = np.log(cierre / cierre.shift(1))
    
    df_rend = pd.DataFrame({
        "rendimiento_simple": rend_simple,
        "rendimiento_log": rend_log,
    })
    return df_rend.dropna()
Estadística Descriptiva

Pruebas de Normalidad: Jarque-Bera y Shapiro-Wilk

Antes de usar VaR paramétrico, verificamos si los rendimientos siguen una distribución normal. En finanzas, casi siempre fallan (colas pesadas).

Jarque-Bera (JB)

$$JB = n \left[ \frac{S^2}{6} + \frac{(K-3)^2}{24} \right]$$

Donde $S$ = asimetría, $K$ = curtosis

H₀: Distribución es normal

Si p-value < 0.05 → rechazamos normalidad (colas pesadas).

Shapiro-Wilk (SW)

Prueba no paramétrica más potente para muestras pequeñas.

H₀: "Los datos provienen de una normal"

Más sensible a desviaciones que JB.

Limitado a max 5000 obs (eficiencia)

services.py - calcular_estadisticas() 
def calcular_estadisticas(self, rendimientos: pd.Series) -> dict:
    # Prueba Jarque-Bera
    jb_stat, jb_pvalue = stats.jarque_bera(rendimientos.dropna())
    
    # Prueba Shapiro-Wilk (máx 5000 obs)
    muestra = rendimientos.dropna()
    if len(muestra) > 5000:
        muestra = muestra.sample(5000, random_state=42)
    sw_stat, sw_pvalue = stats.shapiro(muestra)
    
    return {
        "media": float(rendimientos.mean()),
        "desviacion_std": float(rendimientos.std()),
        "asimetria": float(rendimientos.skew()),
        "curtosis": float(rendimientos.kurtosis()),  # Exceso
        "jarque_bera_pvalue": float(jb_pvalue),
        "shapiro_wilk_pvalue": float(sw_pvalue),
        "es_normal": bool(jb_pvalue > 0.05 and sw_pvalue > 0.05),
    }
ARCH/GARCH

Modelado de Volatilidad Condicional

La volatilidad de activos financieros no es constante — después de choques, aumenta (clustering). ARCH y GARCH capturan esta dinámica.

ARCH (1)

$$\sigma_t^2 = \omega + \alpha_1 \epsilon_{t-1}^2$$

Volatilidad depende del error previo cuadrado.

GARCH (1,1)

$$\sigma_t^2 = \omega + \alpha \epsilon_{t-1}^2 + \beta \sigma_{t-1}^2$$

Añade dependencia de la volatilidad previa (mean reversion).

Uso en RiskLab: GARCHService implementa el modelo para pronosticar volatilidad futura, refinando estimaciones de VaR.

VaR y CVaR

Value at Risk: Tres Métodos

VaR(95%) = pérdida máxima esperada en condiciones normales, con 95% de confianza. En otros términos: "hay 5% probabilidad de perder más."

1. Paramétrico

$$\text{VaR} = \mu + z_\alpha \sigma$$

Asume normalidad. Rápido pero impreciso con colas pesadas.

2. Histórico

Percentil empírico directo de datos pasados. Robusto a distribuciones no normales.

3. Monte Carlo (10,000 simulaciones)

Genera escenarios normales con media y σ históricos, luego calcula rentabilidades sintéticas del portafolio.

CVaR (Expected Shortfall): promedio de pérdidas que exceden VaR → captura riesgo de cola.

Optimización

Markowitz y CAPM

Markowitz: simula miles de portafolios aleatorios, calcula Sharpe, identifica frontera eficiente y portafolios óptimos (máximo Sharpe, mínima varianza).

  • Portfolio return: $R_p = \sum w_i r_i$
  • Portfolio σ: $\sigma_p = \sqrt{w^T \Sigma w}$
  • Sharpe: $\frac{R_p - R_f}{\sigma_p}$

CAPM: calcula Beta por regresión vs benchmark, retorna riesgo esperado = $R_f + \beta(R_m - R_f)$.

  • Beta > 1: activo más volátil que mercado (agresivo)
  • Beta < 1: más defensivo
