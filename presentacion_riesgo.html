BaseSettings y Variables de Entorno

Settings(BaseSettings) carga todas las API keys y parámetros desde el archivo .env — nunca hardcodeados en el código.

  • @lru_cache sobre get_settings(): el .env se lee una sola vez al arrancar (patrón singleton).
  • Cada atributo tiene tipo, valor por defecto y descripción.
  • Parámetros financieros configurables: nivel de confianza VaR, períodos de indicadores, tickers por defecto.
  • CORS: permite que Streamlit (8501) consuma el backend (8000).
Pregunta frecuente: "¿Dónde están las API keys?" → En backend/.env, fuera del repositorio Git. Settings las lee automáticamente por nombre de variable de entorno.

dependencies.py

Inyección de Dependencias con Depends()

Depends() es el sistema de inyección de FastAPI. Las rutas reciben servicios ya construidos en lugar de instanciarlos internamente.

  • ¿Por qué no instanciar en la ruta? Porque la ruta quedaría acoplada a la implementación concreta. Con Depends() es intercambiable y testeable.
  • SettingsDep: inyecta configuración.
  • ValidTickerDep: valida formato del ticker antes de la ruta.
  • FredServiceDep: inyecta el cliente HTTP a FRED ya configurado.
  • HttpClientDep: cliente httpx.AsyncClient reutilizable.
Pregunta: "¿Qué dependencias inyectan con Depends()?"
→ Configuración, servicios de datos, cliente HTTP y validador de ticker. Ningún endpoint contiene lógica de negocio directa.
dependencies.py + main.py

Decoradores

Decoradores Personalizados

El proyecto implementa dos decoradores propios en dependencies.py (requisito Semana 1):

  • @timing_decorator: registra el tiempo de ejecución de funciones costosas (GARCH, Monte Carlo).
  • @cache_result(ttl=300): cachea resultados por N segundos. Evita llamar a yfinance en cada request.

Además, FastAPI usa sus propios decoradores de ruta:

  • @router.get(path): registra función como endpoint GET.
  • @router.post(path): registra función como endpoint POST.
  • response_model=: valida y serializa la respuesta automáticamente.
¿Qué hace @lru_cache? Memoriza el resultado de get_settings(): la primera llamada lee el .env, las siguientes retornan el objeto ya creado sin releer el archivo.
dependencies.py — decoradores propios 
def timing_decorator(func: Callable) -> Callable:
    """Registra tiempo de ejecución (sync y async)."""
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        elapsed = time.perf_counter() - start
        logger.info(f"[TIMING] {func.__name__} → {elapsed:.3f}s")
        return result
    return wrapper

def cache_result(ttl_seconds: int = 300):
    """Cachea resultado con TTL. Evita llamadas repetidas a yfinance."""
    cache: dict = {}
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            key = str(args) + str(kwargs)
            if key in cache:
                result, ts = cache[key]
                if time.time() - ts < ttl_seconds:
                    return result  # Cache hit
            result = func(*args, **kwargs)
            cache[key] = (result, time.time())
            return result
        return wrapper
    return decorator

# Uso en DataService:
@cache_result(ttl_seconds=3600)
def get_info_activo(self, ticker: str) -> dict: ...

Pydantic v2 — Validación de Datos

Cada entrada y salida de la API está tipada y validada. Pydantic lanza 422 Unprocessable Entity automáticamente si algo no cumple las reglas — antes de ejecutar cualquier cálculo.

@field_validator (campo individual)

@field_validator("tickers")
@classmethod
def tickers_mayusculas(cls, v):
    return [t.upper().strip() for t in v]

@field_validator("pesos")
@classmethod
def pesos_positivos(cls, v):
    if any(p < 0 for p in v):
        raise ValueError("Pesos ≥ 0")
    return v

@model_validator (entre campos)

VaRRequest — validación cruzada
@model_validator(mode="after")
def validar_portafolio(self):
    # 1. Misma cantidad de tickers y pesos
    if len(self.tickers) != len(self.pesos):
        raise ValueError(
            "len(tickers) != len(pesos)")

    # 2. Pesos suman 1.0 (±0.01 de tolerancia)
    suma = sum(self.pesos)
    if abs(suma - 1.0) > 0.01:
        raise ValueError(
            f"∑pesos={suma:.4f}, debe ser 1.0")
    return self

async / await

Programación Asíncrona

Los endpoints que hacen I/O de red (llamadas a FRED, yfinance) son async def para no bloquear el servidor mientras esperan respuesta.

  • async def: la función puede ser suspendida y cedida.
  • await: punto exacto donde se suspende y otro request puede avanzar.
  • Beneficio: múltiples clientes simultáneos sin necesidad de threads extra.
  • httpx.AsyncClient: cliente HTTP asíncrono (en lugar de requests síncrono).
Ejemplo práctico: mientras el hilo A espera la respuesta de FRED en /macro, el hilo B puede calcular el VaR para otro usuario. Sin async, el hilo A bloquearía al hilo B.
yfinance es síncrono internamente, pero FastAPI lo detecta y lo corre en un thread pool automáticamente cuando está dentro de una ruta async def.

DataService

Descarga y Limpieza de Datos

DataService es el único punto de contacto con yfinance. Todos los demás servicios consumen datos a través de él.

  • auto_adjust=True: precios ajustados por splits y dividendos.
  • ffill().dropna(): forward fill para huecos de fines de semana; elimina filas incompletas.
  • @cache_result(3600): metadata del activo se cachea 1 hora.
  • @timing_decorator: registra tiempo de descarga en los logs.

Rendimiento log vs simple:

rt = ln(Pt / Pt-1)    vs    Rt = (Pt - Pt-1) / Pt-1

Se prefiere el log porque es aditivo en el tiempo y más estacionario. Por ejemplo: la ganancia de 2 días = r₁ + r₂ (no así con Rt).

Módulo 2 — Rendimientos

Propiedades Empíricas y Pruebas de Normalidad

Antes de usar VaR paramétrico debemos verificar si los rendimientos siguen una normal. En finanzas, casi siempre fallan — tienen colas pesadas (fat tails).

Jarque-Bera (JB)

JB = n · [ S²/6 + (K-3)²/24 ]

Donde S = asimetría, K = curtosis.
H₀: distribución es normal.
p < 0.05 → rechazamos normalidad.

Shapiro-Wilk (SW)

Prueba no paramétrica. Más potente que JB para muestras pequeñas (<5000 obs). Misma H₀.
Usamos muestra aleatoria de 5000 obs para eficiencia.

services.py — calcular_estadisticas()
def calcular_estadisticas(self, rendimientos: pd.Series):
    jb_stat, jb_p = stats.jarque_bera(rendimientos.dropna())

    muestra = rendimientos.dropna()
    if len(muestra) > 5000:
        muestra = muestra.sample(5000, random_state=42)
    sw_stat, sw_p = stats.shapiro(muestra)

    return {
        "media": rendimientos.mean(),
        "desviacion_std": rendimientos.std(),
        "asimetria": rendimientos.skew(),
        "curtosis": rendimientos.kurtosis(), # exceso
        "jarque_bera_pvalue": jb_p,
        "shapiro_wilk_pvalue": sw_p,
        # Normal solo si AMBAS no rechazan H₀ al 5%
        "es_normal": (jb_p > 0.05) and (sw_p > 0.05),
    }

Hechos estilizados de los rendimientos financieros

Curtosis > 3 → colas más pesadas que la normal · Asimetría negativa → caídas más pronunciadas · Volatility clustering → períodos agitados se agrupan. Estos hechos justifican GARCH (Módulo 3) y VaR histórico/Monte Carlo (Módulo 5).

Módulo 3 — ARCH/GARCH

Volatilidad Condicional

La volatilidad no es constante. Después de un shock, aumenta y tarda en volver. GARCH modela la varianza de hoy en función de errores y varianzas pasadas.

ARCH(1)

σ²t = ω + α · ε²t-1

Volatilidad depende solo del shock de ayer. Memoria corta. Simple.

GARCH(1,1) ⭐ más usado

σ²t = ω + α·ε²t-1 + β·σ²t-1

Añade memoria de la varianza anterior. Persistencia = α+β. Si ≈1, la volatilidad tarda mucho en volver a su media.

EGARCH(1,1)

ln(σ²t) = ω + α|εt-1| + γ·εt-1 + β·ln(σ²t-1)

Asimétrico: γ < 0 captura el efecto apalancamiento (caídas ↑ vol más que subidas).

Selección de modelo: AIC y BIC

Criterios de información penalizados por número de parámetros.
Menor AIC/BIC = mejor modelo.
Evita sobreajuste: un modelo con más parámetros siempre mejora el likelihood, pero AIC lo penaliza.

Diagnóstico de residuos

Después de ajustar GARCH, los residuos estandarizados (εt/σt) deben comportarse como una normal estándar.
Se verifica con Jarque-Bera sobre residuos. Si p > 0.05, el modelo capturó bien la estructura.

Módulo 5 — VaR y CVaR

Value at Risk: Tres Métodos

VaR(95%) = pérdida máxima esperada con 95% de confianza en un horizonte dado. Hay 5% de probabilidad de perder más.

1. Paramétrico

VaR = -(μ + z·σ·√h)

z = percentil de la normal (−1.645 para 95%). Rápido, analítico. Subestima riesgo con fat tails.

2. Histórico

VaR = −percentil(r, 5%)

Usa distribución empírica real. No asume ninguna distribución. Robusto pero depende del histórico.

3. Monte Carlo

simula N(μ,σ²) 10,000 veces → percentil 5%

Genera escenarios sintéticos con los parámetros históricos. Flexible, costoso computacionalmente.

CVaR (Expected Shortfall)

Promedio de pérdidas que ya superaron el VaR.
Responde: "cuando se rompe el VaR, ¿cuánto se pierde en promedio?"
Siempre es mayor que el VaR. Más conservador y coherente como medida de riesgo.

¿Qué método recomienda RiskLab?

El backend aplica Jarque-Bera al portafolio:
• p > 0.05 (normal) → recomienda paramétrico
• p < 0.05 (fat tails) → recomienda histórico
La recomendación aparece en el campo mejor_metodo_recomendado del JSON.

Módulo 4 — CAPM

CAPM y Riesgo Sistemático

El CAPM relaciona el riesgo de un activo con su rendimiento esperado. Solo el riesgo sistemático (no diversificable) merece compensación.

E(Ri) = Rf + β × (Rm − Rf)
  • Rf: tasa libre de riesgo → obtenida de FRED (DGS3MO).
  • Rm: rendimiento del benchmark (S&P 500).
  • β: sensibilidad del activo al mercado.
  • β > 1.2 → Agresivo (más volátil que el mercado).
  • 0.8 – 1.2 → Neutro.
  • β < 0.8 → Defensivo (más estable que el mercado).
Beta = Cov(Ri, Rm) / Var(Rm)

Calculado con scipy.stats.linregress. El slope = β, el R² indica qué % del riesgo del activo es sistemático.

services.py — CAPMService 
def calcular_beta(self, rend_activo, rend_mercado):
    df = pd.concat([rend_activo, rend_mercado], axis=1).dropna()

    # Regresión OLS: r_activo = α + β·r_mercado + ε
    slope, intercept, r_val, p_val, _ = stats.linregress(
        df["mercado"], df["activo"]
    )
    return {
        "beta":           slope,       # ← la pendiente ES beta
        "alpha_regresion": intercept,   # Alpha de Jensen diario
        "r_cuadrado":     r_val**2,    # % riesgo sistemático
    }

# R² = 0.75 → 75% del riesgo del activo se explica
# por el movimiento del mercado (sistemático).
# El 25% restante es riesgo propio (diversificable).

# Alpha de Jensen anualizado:
# rendimiento_real_anual − rendimiento_esperado_CAPM
# Si > 0: el activo generó valor adicional al riesgo asumido
Módulo 6 — Markowitz

Frontera Eficiente

Idea central: combinando activos con baja correlación se puede obtener el mismo rendimiento con menor riesgo, o mayor rendimiento con el mismo riesgo.

Rp = Σ wᵢ·rᵢ     σp = √(wᵀ Σ w)
Sharpe = (Rp − Rf) / σp
  • Simula 10,000 portafolios aleatorios con pesos que suman 1.
  • Portafolio de máximo Sharpe: mejor relación rendimiento/riesgo.
  • Portafolio de mínima varianza: menor riesgo posible.
  • La frontera eficiente es el borde superior de la nube de portafolios.
Pregunta: "¿Qué significa diversificar?"
→ Dos activos con correlación ≈ −1 se compensan mutuamente. σp < promedio de σi. El riesgo del portafolio es menor que el riesgo promedio de cada activo por separado.

def construir_frontera(self, tickers, df_rendimientos, rf):
    media = df_rendimientos.mean() * 252  # Anualizar
    cov   = df_rendimientos.cov()  * 252

    for i in range(self.n_portafolios):
        # Pesos aleatorios que suman 1
        pesos = np.random.random(len(tickers))
        pesos = pesos / pesos.sum()

        rend_p = np.dot(pesos, media)
        var_p  = np.dot(pesos, np.dot(cov, pesos))
        sigma_p= np.sqrt(var_p)
        sharpe = (rend_p - rf) / sigma_p

Preguntas Frecuentes del Profesor

¿Dónde están las API keys?

En backend/.env, fuera del repositorio. Settings(BaseSettings) las carga automáticamente. El .gitignore excluye el .env. Solo el .env.example con placeholders va al repo.

¿Qué pasa si yfinance o FRED no responde?

El bloque try/except atrapa el httpx.TimeoutException y lanza HTTPException(503) con mensaje claro. El servidor no cae — retorna un error controlado al cliente.

¿Por qué Depends() y no instanciar en la ruta?

Porque la ruta quedaría acoplada a una implementación concreta. Con Depends() el servicio es intercambiable, testeable por separado y la configuración queda centralizada en las fábricas de main.py.

¿Qué método de VaR es más adecuado?

Depende de Jarque-Bera. Si los rendimientos son normales → paramétrico. Si tienen colas pesadas (casi siempre en acciones) → histórico o Monte Carlo, ya que capturan mejor los eventos extremos reales.

¿Cómo validan datos de entrada?

@field_validator valida cada campo. @model_validator(mode="after") verifica consistencia entre campos (len(tickers)==len(pesos), ∑pesos=1.0). Pydantic lanza 422 antes de ejecutar cálculos.

¿Cuál es la composición del portafolio óptimo?

La retorna el endpoint POST /frontera-eficiente en el campo portafolio_max_sharpe.pesos. Es el portafolio con mayor ratio Sharpe entre los 10,000 simulados. También retorna el de mínima varianza.
